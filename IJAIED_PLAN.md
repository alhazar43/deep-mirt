## Part I: Framing — DEEP-MIRT as a Framework, GPCM as an Instantiation

### Current state (correct in spirit, not in execution)

The abstract and introduction already make the right distinction:
> *"We present DEEP-MIRT, a framework… We instantiate the framework for ordinal responses via a GPCM head, yielding DEEP-GPCM and ATTN-GPCM."*

This is the right argument. The problem is that the claim is **not substantiated** anywhere in the paper.

### What is missing

1. **No framework interface description.** There is no section or paragraph that defines what contract a response head must satisfy — i.e., what inputs it receives (θ_t from summary, item embedding k_t) and what it must output (category probability vector). Without this, "framework" is just a word.

2. **Diagram is GPCM-specific, not framework-level.** Every box in the architecture diagram (GPCM Head, α/β, monotone thresholds) is a GPCM detail. A reviewer will ask: where is the framework? The diagram caption should explicitly say: *"The GPCM Head (green) shows one instantiation of DEEP-MIRT's response head interface; alternative psychometric models (GRM, nominal response model) can replace this component without changing the backbone."*

3. **Contributions conflate framework and instantiation.** Contribution 2 is titled "Interpretable ordinal instantiation via GPCM" — fine. But the contributions list does not separately state what the *framework contribution* is, distinct from the GPCM-specific choices.

### Fix (no new experiments needed)

Add a short paragraph (≈ ½ page) to the Methodology section explicitly describing the DEEP-MIRT interface:
- Any differentiable observation model mapping `(θ_t, k_t) → P(r | θ_t, item)` can slot into the head position.
- Name two or three alternatives: Graded Response Model (cumulative logistic), Nominal Response Model (unordered categories), 2PL/3PL for binary.
- Explain why GPCM is chosen for this study: adjacent-category model, natural for rubric progressions, monotone threshold constraint is educationally meaningful.
- Rewrite Contributions 1–3 to cleanly separate: (1) PKT problem formulation + DEEP-MIRT framework; (2) GPCM instantiation with monotone parameterization and ordinal encodings; (3) SIE for scalable deployment.

## Part II: Critical Assessment of Experiments

### Fatal gaps — paper cannot be submitted without fixing these

**F1. Four experiments are entirely TBD/placeholder.**
- Table 5: Sensitivity analysis (K, imbalance, seq length, sparsity, temperature) — all TBD
- Figure 4: Stability and robustness — placeholder box
- Figure 5: Interpretability case studies — placeholder box
- Table 7: Proxy-ordinality on real data — all TBD

A journal editor will not send a paper with four placeholder experiments to reviewers. These must be completed or explicitly cut from the paper.

**F2. No external baselines in the primary results table (Table 1).**
Section 5.3 describes three baselines — Multiclass DKVMN, Ordinal Regression Head (CORAL-style), Static GPCM — and then none of them appear in any results table. Table 1 compares only three DEEP-MIRT variants against each other. This makes it impossible to evaluate whether any of the proposed models beats a naïve approach. This is the single most damaging gap. IJAIED reviewers will reject on this alone.

**F3. Synthetic-only for the core ordinal claim.**
All completed ordinal results are on data generated by the authors using the same GPCM assumptions as the model. This is circular: the model recovers IRT parameters from data it was designed to fit. Proxy-ordinality (real data) is TBD. IJAIED explicitly expects evaluation on real educational data.

### Serious gaps — likely rejection at review stage

**S1. No standard deviations anywhere.**
Training setup says "we report mean±std across multiple seeds" but Table 1 shows only single point estimates. Differences such as QWK 0.640 vs 0.618 at K=3 are statistically uninterpretable without standard errors. Every recent IJAIED KT paper reports mean±std.

**S2. Ablation section is prose only — no table.**
The ablation section describes results in words: "the three-component combination yields the strongest performance with weights λ_focal=0.2, λ_QWK=0.2, λ_WOL=0.6." There are no numbers for the ablated variants. Reviewers cannot verify any of this.

**S3. Parameter recovery is qualitative only.**
Figure 3 shows scatter plots. The scalability section (Table 4) has r_α and r_β numbers, but the base model recovery analysis (Section 6.3) never states correlation values for K=2,3,4,5 in tabular form. The β threshold compression bias is described but not quantified: how large is it? Does it worsen with K? These should be numerical.

**S4. K=2 row in Table 1 is orphaned.**
The K=2 subtable in Table 1 has no DKT or DKVMN comparison. Binary baselines appear only in Table 2. Either merge K=2 into Table 2 and add DKT/DKVMN there, or add DKT/DKVMN to Table 1 at K=2. Currently the binary row is self-referential.

---

### Moderate gaps — reviewer revision requests

**M1. Binary compatibility results show weakness, framed as strength.**
DEEP-GPCM achieves 0.723 AUC on ASSIST2015 vs 0.729 for DKVMN. This is fine — backward compatibility doesn't require superiority — but saying it's "competitive" when it underperforms on the primary metric will draw reviewer scrutiny. Frame it clearly as a sanity check, not a performance claim.

**M2. Training stability is described but not demonstrated.**
The training setup subsection describes temperature sensitivity and gradient explosions in ATTN-GPCM-TD. Figure 4 (stability) is entirely placeholder. Either complete the stability experiment or remove the discussion of instability from the training setup (since it raises questions the paper then doesn't answer).

**M3. Simulation design is one DGP.**
The synthetic evaluation uses one fixed data-generating process (θ ~ N(0,1), α ~ LogNormal(0,0.3), β ~ sorted N(0,1)). A stronger simulation study would test multiple DGPs — e.g., varying α spread (low vs high discrimination spread), varying β spacing (well-separated vs clustered thresholds), varying sequence length distributions. One DGP limits generalizability claims.
